# 仕様書駆動テストの大規模適用ガイド

本プロジェクト(APIキー発行台帳)での演習結果をもとに、
(1) 大規模アプリへの適用可能性、(2) 試験が作りやすい仕様書かの確認方法、
(3) コードから仕様書をリバースする際のプロンプト、を整理する。

## 1. 大規模アプリへの適用可能性

結論: **技術的には十分可能で効果も出せるが、成功要因はテストコードよりも
「仕様書のテスト可能化」と「アプリのテスト容易性」への投資にある。**

### 今回の演習がうまくいった本当の理由

1. **仕様書がテスト可能な粒度で書かれていた** — EARS記法で境界値(50/51、1,000,000/1,000,001)を一意に規定し、エラー文言を仕様書内で確定させた。バグ検出はテストコードではなく仕様書の手柄。
2. **アプリがテスト容易性を持って作られていた** — 全ウィジェットにobjectName、`--test`起動(一時DB・ウィンドウ位置固定)、`APP_FAKE_NOW`による時刻固定。
3. **範囲が1画面だった** — 32ケースで約70秒。テストごとにアプリを再起動する方式が成立するサイズ。

### 大規模アプリで課題になる点と対処

| 課題 | 対処 |
|---|---|
| 仕様書の品質(曖昧・古い・暗黙知前提) | REQ-ID付きのテスト可能な要求への書き直し。この作業自体が仕様の矛盾・抜けを大量に発見する |
| E2Eだけでは実行時間が破綻(数千ケース→数時間以上) | テストピラミッド化: ロジック網羅(境界値・異常系)はユニット/内部フックの高速テストへ、E2Eは「UIAへの公開確認」と「主要業務フロー」に絞る |
| 既存アプリにテスト容易性がない | objectName付与・テスト用起動モード・時刻/乱数の注入口をアプリ側に後付け(開発チームの協力が不可欠) |
| フレームワーク固有の癖(本件ではQtのAutomationIdフルパス化、コンボ選択パターン等) | 最初に代表的なコントロール種別ごとの操作検証だけを行う技術スパイクを挟む |

### 現実的な進め方

いきなり全面展開せず、**最重要業務フロー1本+画面1〜2枚でパイロット**を回す。
今回と同じ手順(仕様のREQ-ID化 → 画面定義 → テスト設計 → 実装 → 録画付き実行 → バグレポート)を適用し、
「仕様書き直し工数」「テストの安定性」「検出バグ」を測ってから拡大判断する。
本リポジトリの一式(ページオブジェクト、conftest、レポート様式)はテンプレートとして流用可能。

## 2. 「試験が作りやすい仕様書」かどうかの確認方法

### 2-1. チェックリスト(静的レビュー)

各要求文について以下を機械的に確認する。1つでも「いいえ」があれば、その要求はテスト設計時に手が止まる。

| # | 観点 | 確認内容 | 本プロジェクトでの例 |
|---|---|---|---|
| 1 | 一意性 | 入力と期待結果が1対1に決まるか。「適切に」「正しく」等の形容詞で逃げていないか | ×「妥当な予算を受け付ける」→ ○「1〜1,000,000の整数」 |
| 2 | 境界値の帰属 | 境界値そのものがOK側/NG側のどちらに属すか明記されているか | 「50文字は合致しない(OK側)、51文字は合致する(NG側)」と補足で明記 |
| 3 | 文言の確定 | ユーザーに見せるメッセージが一言一句確定しているか(表記ゆれ禁止) | エラーメッセージ一覧(MSG-001〜006)を仕様書内に表で確定 |
| 4 | 観測点 | 期待結果をどこで観測するか(画面のどの欄、DB、ファイル)が特定できるか | 「ステータス欄に表示」+ screen-spec.md の objectName 対応表 |
| 5 | 判定順序 | 複数の検証が同時に成立し得る場合、どれが優先されるか決まっているか | REQ-009 で検証順序を固定し「最初の1件のみ表示」と規定 |
| 6 | 前処理の定義 | トリム・正規化など入力の前処理が定義されているか(全角空白は?等) | 「トリム=半角空白0x20のみ。全角空白は対象外」と用語定義 |
| 7 | 初期状態 | 画面表示直後の各コントロールの状態が規定されているか | REQ-001〜006 |
| 8 | 副作用の網羅 | 成功/失敗それぞれで「何が起きるか」と「何が起きてはならないか」の両方があるか | 「保存する」だけでなく「NG時は保存しない・入力欄を変更しない」(REQ-015/016/022) |
| 9 | ID付与 | 各要求に一意のIDがあり、テストケースから参照できるか | REQ-xxx ⇔ TC-xxx のトレーサビリティ表 |

### 2-2. 動的な確認方法: 「テスト設計できるか」で試す

チェックリストより確実なのは、**実際にテスト設計をやらせてみる**こと。

- **方法A: テストケース化テスト** — 仕様書だけを渡して(コードは見せずに)テスト設計書を書かせる。「期待結果が一意に書けないケース」「仕様書に質問しないと決められない点」が出てきたら、それがそのまま仕様書の欠陥リストになる。今回のプロジェクト構成(テスト担当は docs/ のみ参照、app/ は読まない)はこの分離をルール化したもの。
- **方法B: 曖昧さの敵対的レビュー** — LLMに「この仕様書から、同じ入力に対して2通り以上の解釈ができる箇所をすべて挙げよ」とレビューさせる(プロンプトは§3-4)。
- **方法C: 境界値クイズ** — 仕様書だけを見て「ちょうど50文字は?」「ちょうど1,000,000は?」「-5は『整数でない』と『範囲外』のどちらのエラー?」に即答できるか。答えに詰まる=仕様書の穴。

## 3. コード→仕様書リバース時のプロンプト集

### 3-0. 大原則: 「現状の動作」と「あるべき仕様」を分離する

コードからのリバースで最も危険なのは、**バグまで仕様として正当化してしまう**こと。
本プロジェクトの演習で仕込んだ BUG-001(`>= 50` の off-by-one)のようなバグは、
コードを素直に読むと「50文字以上はエラー」という"仕様"としてリバースされてしまう。

対策として、リバース時は必ず:
- 生成する各要求に「実装からの観測事実」であることを示すタグ(例: `[観測]`)を付け、人間のレビューで `[確定]` に昇格させる
- **境界値比較(`>` vs `>=`)、エラー文言、判定順序は全件「要確認」として人間に列挙提示**させる(バグが最も混入しやすい箇所のため)

### 3-1. ステップ1: インベントリ作成(全体把握)

```text
対象コードベースを読み、仕様書リバースの準備として以下のインベントリを
Markdown表で作成してください。この段階では仕様の記述はしないでください。

1. 画面一覧: 画面名 / 起動経路 / 主要な役割
2. 画面ごとのコントロール一覧: 種別 / 識別子(objectName等) / ラベル文言
3. ユーザー操作イベント一覧: どの操作が、どの処理を起動するか
4. 永続化・外部連携の一覧: テーブル/ファイル/API と、それを読み書きする処理
5. バリデーション処理の一覧: 対象項目 / 検証内容 / エラー時の挙動(場所のみ)
```

### 3-2. ステップ2: 振る舞いのEARS化(画面単位で)

```text
画面「◯◯」について、コードから観測できる振る舞いを要件仕様書に
リバースしてください。ルール:

- EARS記法(Ubiquitous / Event-driven / State-driven / Unwanted behavior)で、
  REQ-001からの連番を付ける
- 1要求=1検証観点。「AかつB」はREQを分割する
- ユーザーに表示される文言(エラー・確認・成功メッセージ)はコードから
  一言一句そのまま転記し、仕様書内に「メッセージ一覧」の表を作る
- 数値・文字数の境界値は、比較演算子(> か >= か)まで確認し、
  境界値そのものがOK側/NG側どちらに属すかを補足として明記する
- 複数のバリデーションが同時に成立し得る場合、コード上の評価順序を
  「検証順序」の要求として明文化する
- 成功時だけでなく、失敗時に「何が起きないか」(保存しない、入力欄を
  変更しない等)も要求として書く
- 画面表示直後の初期状態(各コントロールの値・活性状態)を要求化する
- 各要求の末尾に [観測] タグを付ける(人間レビューで確定させるため)
```

### 3-3. ステップ3: 要確認事項の抽出(バグの仕様化防止)

```text
ステップ2で作成した仕様書ドラフトについて、「実装がバグである可能性」を
疑うべき箇所を優先度付きで列挙してください。特に:

1. 境界値比較: >= と > の取り違えがあり得る箇所すべて
   (例:「50文字以上でエラー」は本当に意図どおりか? 51文字ではないか?)
2. 文言: 同種メッセージ間の表記ゆれ(「ください/下さい」等)、
   誤字の疑いがあるもの
3. 判定順序: 順序に依存した動作のうち、偶然の実装順に見えるもの
4. 副作用: 成功/失敗時の画面状態変化のうち、意図か手抜きか判断できないもの
5. 例外・異常系: catchして握りつぶしている箇所、未処理の入力パターン

各項目に「現在の実装の動作」と「一般的に期待される動作」を併記し、
業務担当者に確認すべき質問文の形にしてください。
```

### 3-4. ステップ4: 仕様書の曖昧さレビュー(テスト可能性の検証)

```text
あなたはテスト設計者です。添付の仕様書「だけ」を情報源として
(コードは参照禁止)、以下を報告してください。

1. 期待結果が一意に決まらない要求: 同じ入力に対して2通り以上の解釈が
   できる箇所を、解釈の例とともにすべて列挙
2. テストケースが書けない要求: 観測点(画面のどこ/DBのどこ)が
   特定できないもの
3. 境界値の帰属が不明な要求: 境界値ちょうどの入力の合否が読み取れないもの
4. 文言が未確定の要求: メッセージの完全一致判定ができないもの
5. 欠落: 初期状態・検証順序・失敗時の副作用禁止など、書かれていない
   ために推測が必要な事項

問題がゼロになるまで仕様書を修正 → 本レビューを繰り返します。
```

### 3-5. 運用のコツ

- **ステップを分けて実行する**: 一度に「コードを読んで仕様書を書け」と指示すると、網羅漏れと実装詳細の混入が増える。インベントリ→EARS化→要確認抽出→曖昧さレビューの4段が安定する。
- **画面単位・機能単位で回す**: 大規模アプリ全体を1回でリバースしない。今回の「1画面=REQ 22件=TC 32件」程度が1バッチの目安。
- **リバースした仕様書 → テスト設計 → 実行**まで通すと、仕様書の穴が実行時エラーとして具体化する。仕様書レビューだけで完結させない。
- **役割分離をルール化する**: 仕様書を書くセッションとテストを書くセッションを分け、テスト側には「docs/ 以外参照禁止」を課す(本プロジェクトの CLAUDE.md 方式)。これにより仕様書の自己完結性が強制的に検証される。
