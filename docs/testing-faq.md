# 実環境(GitLab + exe配布)でのテスト運用 Q&A

社内実環境(GitLabでコード管理、exeでアプリ配布)への適用に関する質問と回答のまとめ。

## Q1. 試験はexeファイルを作成してからになるか?

**すべての試験をexe作成後に回す必要はない。ただし最終確認はexeに対して行うべき。**

### 試験の層とタイミング

| 層 | 対象 | タイミング | 内容 |
|---|---|---|---|
| ユニット/ロジックテスト | ソースコード | コミット/MRごと(exe不要) | 境界値・異常系の網羅。ヘッドレスで数秒〜数分 |
| 内部フックUIテスト(pytest-qt等) | ソースコード | コミット/MRごと(exe不要) | ウィジェット単位の検証。offscreen実行可 |
| E2E UIテスト(本プロジェクトの方式) | ビルド済みexe | ビルド後 | 出荷する成果物そのものへの最終確認 |

### ポイント

- UIA方式のE2Eテストは**起動済みプロセスのウィンドウを外部から操作する**ため、相手が `python -m app` でもexeでも動く。conftest.py の起動コマンドを環境変数(例: `APP_UNDER_TEST=C:\path\to\app.exe`)で切り替え可能にすれば、同じテスト資産を開発中はソース相手に、リリース前はexe相手に使い回せる。objectNameベースのロケータはビルド方法に影響されない。
- **exeに対する試験には固有の価値がある**: リソース同梱漏れ・DLL不足・作業ディレクトリやパスの違いなど、パッケージング起因のバグはexeにしたときにしか発生しない。「テストは出荷するものに対して行う」が原則。
- exeにも `--test` 起動モードや時刻固定(`APP_FAKE_NOW`)が効く状態でビルドする(テスト容易性はビルド後も残す)。

### GitLab CIパイプラインの典型形

```yaml
stages: [test, build, e2e]

unit-test:        # コミットごと・exe不要・高速
  stage: test
  script: pytest tests/unit --junitxml=report.xml

build-exe:
  stage: build
  script: pyinstaller app.spec   # など
  artifacts:
    paths: [dist/app.exe]

e2e-ui-test:      # ビルド成果物に対して実行
  stage: e2e
  tags: [windows-desktop]        # 専用Windowsランナー
  script: pytest tests/e2e --junitxml=report.xml
  variables:
    APP_UNDER_TEST: dist/app.exe
  artifacts:
    when: always
    paths: [artifacts/]          # スクリーンショット・録画
    reports:
      junit: report.xml          # MR画面に結果を表示
```

注意: GUIのE2EテストはランナーにデスクトップセッションのあるWindows環境が必要
(UIAは画面が実在しないと動かない)。GitLab RunnerをWindowsサービスとして入れると
デスクトップなしで動くため、自動ログオンのVM/実機でコンソールセッション上に
ランナーを常駐させる構成が定番。

## Q2. C#/C++のアプリはexeを作らずに起動できるか?

厳密にはできないが、問題にはならない。C#/C++はコンパイル言語のため実行には必ずビルド(exe/dll生成)が必要。区別すべきは次の2つ。

- **開発ビルド**: `dotnet run` やVisual Studioのビルドは数秒〜数分で自動的にexeを作って起動する。日常開発ではこれで十分
- **配布用パッケージ(インストーラ)**: 時間がかかるのはこちら

「ロジック検証は開発ビルドで高速に回し、配布用パッケージは最終確認だけ」という2段構えはC#/C++でもそのまま成立する。ユニットテストはアプリ全体をビルドせず、対象クラス/ライブラリ単位でビルドして実行できるためさらに速い。

## Q3. GitLab CIのパイプラインで自動で試験できるか?

できる。それがCIの本来の目的。`git push` やマージリクエスト(MR)作成をトリガーに、ビルド→試験→結果報告まで人手なしで実行される。

- JUnit XMLを出力すればMR画面に失敗テストが表示される
- 「試験が通らないとマージ不可」のルール設定が可能
- 時間のかかる試験は毎晩の定時実行(スケジュールパイプライン)に置くのが定番

## Q4. 試験を実施するPCは固定になるか?

**GUI E2E試験は実質的に固定(専用機)を推奨。**

- GitLab Runnerを登録したWindows機で、デスクトップセッションが常にある状態(自動ログオン)が必要
- 画面解像度・DPIスケーリング・OSバージョンの違いはUIテストの安定性に影響するため、環境固定自体に価値がある
- 1台に限る必要はなく、同構成のVMを複数並べて並列実行も可能(VMテンプレート化が理想)
- ユニットテストは任意のマシンで動くため固定不要

## Q5. オフライン環境のPCでも試験は可能か?

**試験の実行自体はインターネット不要。** pytest + uiautomation + アプリの実行はローカルで完結する。事前準備として Python本体・パッケージ(wheelを持ち込みオフラインインストール)・ffmpeg 等を導入しておく。

注意点:

- **GitLab CI連携**にはGitLabサーバーへの接続が必要。社内イントラのセルフホストGitLabなら閉域網内で完結できる。完全スタンドアロン機ならタスクスケジューラ等でローカル実行し結果を手動回収する形になる
- **Claude Codeでの試験作成・修正**はインターネット(Anthropic API)が必要。「テストの作成・保守はオンライン環境、実行はオフライン環境」という分担になる

## Q6. インストールの試験も可能か?

可能。「デプロイメントテスト」として確立された分野。ほとんどのインストーラはサイレントインストールに対応している(MSI: `msiexec /i app.msi /qn`、NSIS: `/S` など)。

自動化の流れ:

1. クリーンな環境にサイレントインストール
2. ファイル・レジストリ・ショートカットの配置を検証
3. インストールされたアプリを起動しスモークテスト(E2E資産を流用)
4. アンインストールして残骸がないことを検証

時間がかかるためコミットごとではなく**夜間の定時パイプライン**に置く。「クリーンな環境」を毎回用意するため、試験前にVMスナップショットを巻き戻す運用とセットで組むのが定石。

## Q7. 試験の待ち時間が長いとClaudeの消費トークンは増えるか?

**増えない。** トークンはテキストの入出力量に対する消費で、経過時間には課金されない。テスト実行中のClaudeはツール完了を待っているだけで、トークンを消費するのは返ってきた実行結果テキストを読むとき。

トークン量に効くのは時間ではなく**出力の長さ**。節約は「試験を速くする」より「出力を簡潔にする」が有効:

- pytest を `-q`(quiet)で実行し、失敗時だけ詳細を出す
- 長大なログはファイルに落とし、Claudeには要約・失敗分のみ読ませる
- 定常運用では `pytest -q --junitxml=...` とし、失敗時だけ深掘りする形がトークン効率が良い
